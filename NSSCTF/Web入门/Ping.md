# [LitCTF 2023]Ping
题目地址：https://www.nssctf.cn/problem/3873  

题目让输入一个 IP 地址，然后点击“PING”按钮，后端会执行系统命令 ping 你输入的IP，并把结果返回给你。
先输入127.0.0.1（本机回环地址），正常返回 ping 的结果，说明这个功能确实存在。  

<img width="692" height="668" alt="image" src="https://github.com/user-attachments/assets/2a287155-1d3e-40ed-ba0d-60cd808c2ea3" />  

再输入127.0.0.1; cat /flag，
>在 Linux 中，分号可以分隔多个命令，依次执行。
所以 ping 127.0.0.1; cat /flag 如果被执行，系统会先执行 ping 127.0.0.1，再执行 cat /flag，后者会读取根目录下的 flag 文件。  

出现弹窗“敢于尝试已经是很厉害了，如果是这样的话，就只能输入ip哦”，说明前端 JavaScript 拦截了输入，它检测到你输入的不是纯 IP 地址（因为里面有 ; 和 /），直接弹出警告，根本没有把请求发给后端。  

<img width="693" height="258" alt="image" src="https://github.com/user-attachments/assets/77387f67-cda8-4091-a979-b7007bdf8768" />  

弹窗提示是前端 JS 做的，它只是客户端的防御，非常脆弱。

按 F12 打开开发者工具，在“设置”里**禁用 JavaScript**（或者用浏览器插件临时禁用），就等于废掉了这个前端检查。  

<img width="692" height="432" alt="image" src="https://github.com/user-attachments/assets/05bfba93-710e-49c4-a61c-8ba7e7f9ef7b" />  

此时，再次输入 127.0.0.1; cat /flag，请求会直接发送到后端服务器，前端不再拦截。  
后端收到你的输入后，没有做任何过滤，直接把整个字符串拼接到系统命令中执行：
`ping 127.0.0.1; cat /flag`得到flag。

## 总结：这道题的完整逻辑

| 步骤 | 做了什么 | 为什么这么做 |
|------|----------|--------------|
| 输入 `127.0.0.1` | 测试功能是否正常 | 确认存在命令执行入口 |
| 输入恶意 payload | 尝试注入命令读取 flag | 想利用 `;` 执行额外命令 |
| 弹窗提示 | 前端 JS 检测到非法字符，拦截请求 | 说明只有前端验证，后端可能没过滤 |
| 禁用 JavaScript | 绕过前端验证 | 让恶意 payload 直接到达后端 |
| 再次输入 payload | 后端执行了 `ping` + `cat /flag` | 后端没有对输入做安全处理，导致命令注入成功 |

---

## 为什么要禁用 JavaScript？
因为这道题的防御**只在浏览器端**，没有在后端做任何过滤。禁用 JS 后，前端检查形同虚设，你的恶意输入就能“骗”过后端执行。**前端校验≠安全校验**，真正的安全必须放在后端。

---

## 命令注入的原理
- 系统执行命令时，如果直接把用户输入拼接到命令字符串里，攻击者可以用 `;`、`&&`、`|` 等符号插入自己的命令。
- 这道题里，后端代码可能类似：
  ```php
  system("ping " . $_POST['ip']);
  ```
- 当你输入 `127.0.0.1; cat /flag`，实际执行的就是：
 `ping 127.0.0.1; cat /flag`
- 所以 flag 就被读出来了。

---

## 两个核心点：
1. **前端限制可以绕过**（比如禁用 JS、修改 HTML、改请求包）。
2. **命令注入漏洞的根本原因**是后端直接把用户输入拼接到系统命令中，而没有过滤或转义。

**前端验证的局限性**和**命令注入的基本利用方式**。以后再遇到类似的输入框，先试试 `; ls` 或 `| whoami`，观察反应，然后再考虑绕过前端。
