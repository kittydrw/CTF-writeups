
### 第一章：Flask 

-   **是什么**：Python写的轻量级Web框架。主打**灵活**和**自由**。
-   **类比**：**毛坯房**。只给你必备的水泥墙（路由和模板），装修（数据库、登录等）全凭自己选。
-   **核心组件**：
    -   **路由**：`@app.route('/')`，负责把网址映射给对应的处理函数。
    -   **模板**：渲染HTML页面。

---

### 第二章：Flask Session 核心原理（重点！）

#### 1. 存储位置（最大的特点）
-   **默认存储在客户端（Cookie中）**。服务器不存Session数据。
-   表现形式：一串看似乱码的字符串（如 `eyJ1c2VyIjoiYWRtaW4ifQ.ZG1haw.abc123`）。

#### 2. 数据结构（三重结构）
Session由三部分用 `.` 拼接而成：

-   **第1段 (数据)**：`eyJ1c2VyIjoiYWRtaW4ifQ`。这是`{"user":"admin"}` 经过 **Base64编码** 后的结果。
-   **第2段 (时间戳)**：防过期用的。
-   **第3段 (签名)**：由 `第1段 + 第2段 + SECRET_KEY` 进行哈希（HMAC）计算得出。**作用是防篡改**。

#### 3. 核心机制：“防君子不防小人”
-   用户可以**解码**查看第1段的内容（因为只是Base64，没有加密）。
-   但用户**不能篡改**数据，因为一旦改了，服务器重算签名，发现和第三段对不上，就会拒绝该Cookie。
-   **防守命门**：这一切的安全，全系于 **`SECRET_KEY`（密钥）** 的保密性。

---

### 第三章：漏洞根源（为什么能伪造？）

当 `SECRET_KEY` 泄露时，攻击者可以做两件事：

1.  解码合法Cookie，摸清数据格式（比如字段叫 `user` 还是 `role`）。
2.  自己改数据（比如把 `guest` 改 `admin`），然后用自己的密钥重新计算签名。
3.  服务器收到后验签通过，**完全信任**，攻击成功。

> **结论**：拿到 `SECRET_KEY` 就等于拿到了上帝权限。

### 第四章：CTF 军火库（必装工具）

**主武器：`flask-unsign`（Python3）**
这是最全能的瑞士军刀，集解码、爆破、签名为一体。

```bash
# 1. 安装
pip3 install flask-unsign

# 2. 解码（查看数据结构，推断字段名）
flask-unsign --decode --cookie "<你的Cookie>"

# 3. 爆破（当密钥隐藏时，猜弱口令）
flask-unsign --unsign --cookie "<你的Cookie>" --wordlist <字典路径>

# 4. 签名（伪造！！！核心操作）
flask-unsign --sign --cookie "{'user':'admin'}" --secret "<泄露的密钥>"
```

**备用武器：`flask_session_cookie_manager3.py`**
> 注意：使用 `-t` 参数时，**必须用双引号包住字典**（如 `-t "{'user':'1'}"`），否则会报错。

### 第五章：CTF 通用解题步骤

遇到 Flask Session 题目，按以下顺序 **无脑梭哈**：

-   **Step 1：信息收集（找密钥）** —— 最耗时的一步！
    -   **F12看源码**：注释里有没有硬编码？（`<!-- secret key is nssctf -->`）
    -   **目录扫描**：找 `.git` 泄露、`www.zip` 备份、`config.py` 文件。
    -   **报错信息**：触发报错（如访问不存在路由），看Debug模式是否开启，有时报错会直接打印环境变量。
    -   **弱口令爆破**：字典尝试 `secret_key`、`password`、`123456` 等。

-   **Step 2：解码（看数据结构）**
    -   用 `flask-unsign --decode` 查看普通用户的Cookie。
    -   找出身份标识字段（常见如 `user_id`、`name`、`role`、`is_admin`）。

-   **Step 3：伪造（生成新Cookie）**
    -   将找到的字段值修改为高权限（如 `user_id` 从 `2` 改 `1`，`name` 改 `admin`）。
    -   用 `flask-unsign --sign` 生成新的Cookie。

-   **Step 4：重放（拿Flag）**
    -   用浏览器Cookie替换，或使用Burp/脚本请求 `/flag`、`/admin` 等接口，收工！

---

### 第六章：避坑指南

1.  **解码只取第一段**：很多人直接把整个Cookie（含点号）拿去Base64解码，这是错的！解码出来全是乱码。**正确姿势**：只取第一个 `.` 之前的内容补上 `=` 号（Base64填充）再解码。
2.  **单引号与双引号**：在Linux命令行中，`--cookie` 后面的JSON格式数据**外层用单引号，内层用双引号**，例如 `'{"name":"admin"}'`。
3.  **区分引擎**：Flask 1.x 和 2.x 的签名算法略有差异，但 `flask-unsign` 通杀。

### 总结
**Flask Session** = 浏览器存数据 + 密钥防篡改。  
**CTF解题** = 找密钥 -> 改管理员 -> 重新签名 -> 浏览器替换 -> 出Flag。  
**找到密钥`SECRET_KEY`，这把CTF你就拿下一半了！**
